# 第 21 章 日志与指标系统

> "Without data, you're just another person with an opinion." —— W. Edwards Deming

一个没有可观测性的系统就像一辆没有仪表盘的汽车——即使性能卓越，驾驶者也无法知道油量、速度和发动机状态。Firecracker 的日志与指标系统为运维人员提供了观察 VMM 内部状态的窗口，同时严格遵循最小开销原则，确保可观测性不会成为性能的累赘。

## Logger 实现

Firecracker 实现了自定义的日志系统而非使用 Rust 生态中现成的日志框架（如 `env_logger` 或 `tracing`），原因在于对二进制大小和运行时开销的严格控制（`// src/vmm/src/logger/`）。

Logger 的核心是一个全局静态实例，通过 Rust 的 `log` crate 提供的标准日志门面（facade）接口对外暴露。这意味着 VMM 中的各个模块使用标准的 `log::info!`、`log::error!` 等宏来记录日志，不需要直接依赖 Firecracker 的 Logger 实现——这种解耦使得日志后端可以在不修改调用方代码的情况下被替换。

日志输出目标通过 API 配置，支持写入文件或 FIFO（命名管道）。FIFO 模式特别适合与外部日志收集系统集成——日志消费者从管道的另一端读取，实现了生产者与消费者的解耦。如果 FIFO 的读取端未连接，写入操作不会阻塞 VMM 主线程，而是丢弃日志条目——这一设计选择优先保证了 VMM 的运行连续性。

为什么不使用 `tracing`？因为 `tracing` 框架引入了显著的编译时和运行时开销，包括 span 创建、subscriber 调度和异步收集器。对于 Firecracker 这种每一毫秒都经过精确预算的系统，这些开销是不可接受的。自定义 Logger 的代码量不足千行，但完全满足需求。

## 日志级别与过滤

Firecracker 支持标准的五级日志级别：Error、Warn、Info、Debug 和 Trace。生产环境通常配置为 Info 或 Warn 级别，开发和调试场景使用 Debug 或 Trace。

日志级别过滤在编译时和运行时各有一层。编译时通过 Rust 的 `log` crate 特性门（feature gate），可以在 release 构建中完全移除 Debug 和 Trace 级别的日志代码，消除它们的运行时开销——包括字符串格式化和参数求值。运行时过滤则通过配置的日志级别阈值进行，低于阈值的日志调用在比较级别后立即返回，开销只有一次整数比较。

这种双层过滤机制的设计哲学是：开发时保持丰富的日志能力以方便调试，发布时通过编译期消除确保零开销。

## METRICS 静态实例

与日志系统类似，Firecracker 的指标系统也围绕一个全局静态实例 `METRICS` 展开（`// src/vmm/src/logger/metrics.rs`）。这个实例在 VMM 启动时初始化，此后所有指标更新操作都通过它进行。

`METRICS` 的设计采用了扁平化的结构体而非动态注册的指标集合。每个需要追踪的指标都是结构体中的一个字段，在编译时确定。这种设计虽然不如动态注册灵活，但带来了显著的优势：无需哈希表查找、无需字符串比较、无需动态内存分配——指标更新操作被简化为对固定内存地址的原子操作，开销接近于零。

为什么选择静态结构体而非像 Prometheus 客户端库那样的动态注册模型？因为 Firecracker 的指标集合在编译时就已确定——不会在运行时动态创建新的指标类型。静态模型在这一约束下是最高效的选择，也消除了指标名称拼写错误等运行时错误的可能。

## 指标类型

Firecracker 的指标体系包含以下几种基本类型：

**计数器（Counter）**：单调递增的累积量，用于记录事件发生次数。例如 API 请求总数、VM exit 次数、virtio 设备中断次数等。计数器使用 `AtomicU64` 实现，保证多线程环境下的更新安全（`// src/vmm/src/logger/metrics.rs`）。`fetch_add` 原子操作确保并发更新不会丢失。

**计量器（Gauge）**：可增可减的瞬时值，用于记录当前状态。例如当前活跃的 vCPU 数量、已分配的 guest 内存大小等。Gauge 同样使用原子类型实现，但支持 `store` 操作来设置任意值。

**直方图类数据**：对于需要了解分布特征的指标（如 I/O 延迟），Firecracker 采用简化的方式记录——不是维护完整的直方图桶，而是记录最小值、最大值和总和，由外部系统进行进一步的统计分析。这种简化再次体现了 VMM 内部最小化开销的原则。

每种指标类型的更新操作都被设计为无锁的。在单线程事件循环中，大部分指标更新实际上不存在并发竞争，原子操作的使用是为了保证 vCPU 线程等少数并发场景下的正确性。

## 指标序列化

Firecracker 将所有指标序列化为 JSON 格式输出。序列化过程发生在指标刷新时——整个 `METRICS` 结构体被遍历，每个字段的名称和当前值被格式化为 JSON 键值对。

JSON 格式的选择基于通用性考量。几乎所有的监控系统（Prometheus、CloudWatch、Datadog 等）都能直接或通过简单转换解析 JSON 格式的指标数据。虽然二进制格式（如 Protocol Buffers）的序列化开销更低，但 JSON 的人类可读性在调试时极为宝贵，且指标刷新频率不高（通常每秒或每几秒一次），序列化开销在可接受范围内。

序列化后的 JSON 文档结构按子系统组织：

```json
{
  "api_server": { "process_startup_time_us": 1234, "sync_response_fails": 0 },
  "block": { "read_count": 512, "write_count": 128 },
  "net": { "rx_bytes_count": 65536, "tx_bytes_count": 32768 },
  "vcpu": { "exit_io_in": 1024, "exit_io_out": 2048 }
}
```

这种层次化结构使得运维人员可以快速定位特定子系统的状态，也便于监控系统按子系统维度聚合和告警。

## 指标刷新机制

指标的刷新（flush）是将内存中的指标快照写出到外部存储的过程。Firecracker 支持两种刷新目标：文件和 vsock。

**文件刷新**：指标被追加写入指定的文件或 FIFO。与日志输出类似，FIFO 模式适合与外部系统实时集成。文件模式则适合离线分析。

**定时刷新**：Firecracker 通过事件循环中的定时器触发定期刷新。刷新间隔通过 API 配置，默认值通常为 1 秒。每次刷新时，`METRICS` 中的所有计数器值被读取并重置为零，这意味着输出的是增量值而非累计值——这简化了外部监控系统的计算逻辑。

为什么使用增量值而非累计值？因为在 microVM 的生命周期管理中，VM 可能频繁地创建和销毁，累计值在 VM 销毁后就失去了意义。增量值可以被外部系统直接聚合，更适合 serverless 场景。

## 结构化日志设计

Firecracker 的日志输出采用结构化格式而非自由文本。每条日志包含固定的字段：时间戳、日志级别、模块路径和消息内容。这种结构化设计使得日志可以被机器解析和索引，支持高效的日志搜索和告警。

日志时间戳使用 UTC 时区的微秒精度时间，避免时区转换带来的混淆。模块路径直接映射到 Rust 的模块系统（如 `vmm::devices::virtio::net`），使得开发者能快速定位日志来源的代码位置。

在生产环境中，Firecracker 的日志通常被收集到集中式日志系统（如 CloudWatch Logs），配合指标数据进行关联分析。例如当某个指标出现异常时，运维人员可以根据时间范围查询对应的日志，快速定位问题根因。

## 可观测性最佳实践

基于 Firecracker 的设计经验，以下可观测性实践值得借鉴：

**指标命名规范**：使用下划线分隔的描述性名称，包含度量单位后缀（如 `_us` 表示微秒，`_count` 表示次数，`_bytes` 表示字节数）。一致的命名约定降低了理解成本。

**错误路径日志**：每个错误处理分支都应记录足够的上下文信息。Firecracker 在错误日志中包含错误码、触发条件和建议的排查方向——在无法 SSH 进入 microVM 内部调试的场景下，日志是唯一的诊断手段。

**指标覆盖关键路径**：每个可能影响性能或可用性的代码路径都应有对应的指标。Firecracker 对 API 处理、设备 I/O、VM exit 等关键路径都设置了延迟和计数指标。

**开销预算**：可观测性代码的 CPU 开销不应超过总工作负载的 1%。Firecracker 通过原子操作、编译期消除和低频刷新来确保这一预算不被突破。

## 本章小结

Firecracker 的日志与指标系统体现了"够用即可"的设计哲学——不追求框架的通用性和功能的丰富性，而是在满足可观测性需求的前提下，将运行时开销压缩到极致。全局静态实例、原子操作、编译期消除和结构化输出共同构成了一个轻量而实用的可观测性方案。在 serverless 场景下，数万个 microVM 的日志和指标汇聚在一起，构成了对整个平台健康状态的全景视图——这正是可观测性的终极价值。

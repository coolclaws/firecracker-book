# 第 18 章 seccomp 过滤器

> "Security is always excessive until it's not enough." —— Robbie Sinclair

如果说 jailer 是 Firecracker 的外层城墙，那么 seccomp 过滤器就是守在每个线程门前的哨兵。即使攻击者突破了 guest 隔离进入 VMM 进程空间，seccomp 也能将其可利用的系统调用限制在极小范围内，从而大幅削弱攻击能力。

## seccomp-BPF 原理

seccomp（secure computing mode）是 Linux 内核提供的系统调用过滤机制。在其最严格的模式下（strict mode），进程只能调用 `read`、`write`、`exit` 和 `sigreturn` 四个系统调用。这显然过于受限，因此 Linux 3.5 引入了 seccomp-BPF 模式，允许用户通过 BPF（Berkeley Packet Filter）程序定义自定义的系统调用过滤规则。

BPF 程序是一种运行在内核虚拟机中的字节码程序。当进程发起系统调用时，内核会先执行附加的 BPF 程序，根据返回值决定允许、拒绝还是终止进程。BPF 程序可以检查系统调用号和参数，但出于安全考虑，不能解引用指针参数——这意味着它只能检查整数类型的参数值。

为什么选择 BPF 而不是其他过滤方式？因为 BPF 运行在内核态，开销极低（通常只有几百纳秒），且其执行模型经过数十年验证，不存在图灵完备性带来的停机问题——BPF 程序必须是有向无环图，保证终止。

## seccompiler crate 设计

Firecracker 将 seccomp 过滤逻辑封装为独立的 `seccompiler` crate（`// src/seccompiler/`）。这个 crate 的核心职责是将人类可读的过滤规则编译为内核可执行的 BPF 字节码。

设计上，`seccompiler` 采用了编译器的经典分层架构：

**前端**负责解析 JSON 格式的过滤规则定义文件。JSON 格式的选择是为了让安全工程师能直接阅读和修改规则，而不需要理解 BPF 字节码。每条规则指定一个系统调用名称、可选的参数约束条件，以及匹配时的动作（允许或拒绝）。

**中间表示**将解析后的规则转换为结构化的 Rust 数据结构。这一层处理系统调用名称到编号的映射（不同架构的系统调用号不同），以及规则之间的冲突检测。

**后端**将中间表示编译为 BPF 指令序列（`// src/seccompiler/src/backend.rs`）。每条 BPF 指令是一个固定大小的结构体，包含操作码、跳转偏移和立即数。后端需要精确计算跳转偏移量，因为 BPF 使用相对跳转。

这种分层设计的好处是显而易见的：如果未来需要支持 YAML 格式的规则定义，只需替换前端；如果需要支持新的 CPU 架构，只需调整系统调用号映射和后端的字节序处理。

## JSON 过滤规则定义

Firecracker 的 seccomp 过滤规则以 JSON 文件形式存储（`// resources/seccomp/`）。一个典型的规则条目如下所示：

```json
{
  "syscall": "write",
  "args": [
    { "index": 0, "type": "dword", "op": "eq", "val": 1 }
  ],
  "action": "allow"
}
```

这条规则表示：允许 `write` 系统调用，但仅当第一个参数（文件描述符）等于 1（标准输出）时。这种参数级别的细粒度控制，使得 Firecracker 能在允许必要操作的同时，最大限度地收紧权限。

规则文件按线程角色组织——VMM 主线程、vCPU 线程和 API 线程各有不同的过滤规则，因为它们需要的系统调用集合不同。这种按角色分离的方式体现了最小权限原则的精细应用。

## 白名单策略

Firecracker 采用白名单（allowlist）而非黑名单（denylist）策略：默认拒绝所有系统调用，只显式允许必需的系统调用。这一策略选择至关重要。

黑名单策略需要预见所有危险的系统调用并逐一禁止，而 Linux 内核不断添加新的系统调用（截至 2024 年已超过 450 个），任何遗漏都可能成为攻击入口。白名单策略则相反——新增的系统调用默认被拒绝，只有经过安全审查后才会被显式加入白名单。

Firecracker VMM 主线程的白名单通常只包含约 30-40 个系统调用，相比 Linux 提供的全部系统调用，攻击面缩减了约 90%。这些被允许的系统调用包括基本的文件 I/O（`read`、`write`）、内存管理（`mmap`、`munmap`）、epoll 操作、KVM ioctl 等 VMM 运行所必需的最小集合。

### 设计取舍

白名单策略的代价是维护成本：每当 Firecracker 引入新功能或升级依赖库，可能需要新的系统调用，此时白名单必须同步更新，否则 VMM 会在运行时被 seccomp 终止。黑名单策略则不存在这一问题——新增的系统调用默认被允许，功能不会中断。然而 Firecracker 团队认为这一维护成本是值得承受的：白名单的遗漏会立即以 VMM 崩溃的形式暴露（fail-closed），而黑名单的遗漏则静默地留下安全漏洞（fail-open）。在安全关键系统中，fail-closed 始终优于 fail-open。此外，将 seccomp 规则定义为独立的 JSON 文件而非硬编码在源码中，降低了更新规则的门槛——安全工程师可以独立审查和修改过滤规则，无需理解 VMM 的 Rust 代码。

## 每线程过滤器安装

seccomp 过滤器通过 `prctl(PR_SET_SECCOMP, ...)` 或 `seccomp(SECCOMP_SET_MODE_FILTER, ...)` 系统调用安装，且只对当前线程生效。Firecracker 利用这一特性，为不同线程安装不同的过滤器。

安装时机也经过精心设计（`// src/vmm/src/builder.rs`）。过滤器在线程完成所有初始化工作之后、进入主循环之前安装。这样做的原因是初始化阶段可能需要额外的系统调用（如 `open` 打开设备文件、`socket` 创建通信通道），而一旦初始化完成，这些系统调用就不再需要。过早安装过滤器会阻止正常初始化，过晚安装则留下攻击窗口。

此外，seccomp 过滤器具有不可逆性——一旦安装，不能被移除或放宽，只能进一步收紧。这一内核级保证意味着即使攻击者获得了进程内的任意代码执行能力，也无法解除 seccomp 约束。

## 架构特定的过滤器

系统调用号在不同 CPU 架构之间不同——例如 `write` 在 x86_64 上是 1，在 aarch64 上是 64。`seccompiler` 必须处理这一差异。

Firecracker 为 x86_64 和 aarch64 分别维护过滤规则文件（`// resources/seccomp/aarch64/` 和 `// resources/seccomp/x86_64/`）。虽然大部分规则在语义上相同，但某些架构特有的系统调用需要单独处理。例如 aarch64 没有 `open` 系统调用（统一使用 `openat`），而 x86_64 两者都有。

`seccompiler` 在编译时根据目标架构选择正确的系统调用号映射表，确保生成的 BPF 字节码在目标平台上正确运行。这一过程对规则编写者透明——他们只需要使用系统调用名称，架构适配由工具链自动完成。

## 过滤器验证与测试

seccomp 过滤器的正确性至关重要——过于宽松会留下安全漏洞，过于严格则导致 VMM 无法正常运行。Firecracker 采用多层验证策略。

单元测试层面，`seccompiler` 验证 BPF 字节码的正确性，包括跳转偏移计算、字节序处理和指令编码。集成测试层面，Firecracker 在 CI 环境中以完整的 seccomp 过滤器运行功能测试，确保所有正常操作路径不被过滤器阻断（`// tests/integration_tests/security/`）。

此外，Firecracker 团队使用 `strace` 跟踪 VMM 运行时实际调用的系统调用，将结果与白名单对比，识别多余的或缺失的规则。这种基于实际运行行为的验证方法，弥补了静态分析可能遗漏的边界情况。

## 纵深防御中的角色

seccomp 不是 Firecracker 安全模型的唯一屏障，而是纵深防御（defense in depth）策略中的关键一层。在 Firecracker 的安全架构中，防御层次从外到内依次为：

jailer 提供 namespace 和 cgroup 隔离，限制 VMM 进程的资源访问范围。KVM 提供硬件辅助的 guest 隔离，防止 guest 直接访问 host 资源。seccomp 限制 VMM 进程可用的系统调用集合，即使前两层被突破，攻击者的能力仍被大幅削弱。

这种分层设计意味着任何单一层面的漏洞都不足以完全攻破系统。攻击者需要同时突破 KVM 的硬件隔离、绕过 seccomp 的系统调用过滤、并逃逸 jailer 的命名空间限制——这在实践中极为困难。

## 本章小结

seccomp 过滤器是 Firecracker 安全架构中不可或缺的组成部分。通过 `seccompiler` crate，Firecracker 将安全策略从 BPF 字节码提升为人类可读的 JSON 规则，降低了维护成本和出错概率。白名单策略、每线程安装、架构适配和多层验证共同构成了一个严密而实用的系统调用过滤方案。理解 seccomp 的设计，有助于我们认识到安全不是单点突破的银弹，而是多层叠加的系统工程。
